Во второй главе будут рассмотрены проектные решения, на основе которых планируется реализовать веб-сервис подбора образов и управления личным гардеробом. Этот раздел нужен для описания структуры будущей системы, ее функций, состава модулей, организации данных, логики подбора и устройства интерфейса. Пользователь будет самостоятельно наполнять цифровой гардероб, а подбор образов будет строиться на алгоритме, основанном на наборе правил. Далее будут последовательно рассмотрены функциональная схема сервиса, архитектура клиентской и серверной частей, структура базы данных, логика алгоритма подбора образов и проект пользовательского интерфейса.
2.1. Проектирование функциональной схемы веб-сервиса
Проектирование функциональной схемы требуется для того, чтобы определить состав действий, которые должна выполнять система, порядок их выполнения и передачу данных между пользователем, интерфейсом, серверной частью и базой данных. Работа сервиса будет строиться как последовательность связанных операций, начиная с заполнения гардероба и заканчивая получением и сохранением подобранного образа.
Функциональная схема должна отражать основной сценарий работы системы и вспомогательные процессы, которые потребуются для полноценного использования сервиса. 
2.1.1. Состав основных функций системы
В системе будет выделено пять взаимосвязанных функций:
регистрация, вход и доступ к персональным данным пользователя;
ведение цифрового гардероба: добавление карточек вещей, загрузка изображения, заполнение характеристик одежды, а также изменение или удаление записи;
автоматический подбор образов по заданным условиям;
сохранение удачных вариантов для последующего просмотра;
формирование краткой аналитики по составу гардероба.
Центральное место в системе планируется отвести подсистеме подбора образов. При этом ее работа будет зависеть от сведений, которые пользователь заранее внесет в гардероб и профиль. Результатом работы этой подсистемы должны стать сформированные варианты образов, доступные для просмотра, оценки и сохранения. По этой причине функциональная схема будет строиться вокруг общего цикла, который включает накопление данных, задание условий, генерацию вариантов, оценку результатов, сохранение выбранных решений и их дальнейшее использование в соответствии с Рисунком 2.1.

2.1.2. Пользовательские роли и сценарии взаимодействия
В проектируемой версии сервиса предполагается одна основная роль, зарегистрированный пользователь. Сначала он должен создать учетную запись и заполнить исходные сведения. После этого он сможет перейти к наполнению цифрового гардероба и добавить информацию о своих вещах. Лишь после этого сервис станет пригоден для подбора образов.
Основной пользовательский сценарий планируется выстроить следующим образом. Пользователь будет открывать страницу генерации, задавать тип события, при необходимости выбирать стиль, желаемые цвета, погодные условия и дополнительные ограничения. Затем запрос будет отправляться на сервер. Серверная часть должна получить из базы данных набор вещей текущего пользователя, дополнить контекст недостающими параметрами и передать данные в модуль подбора. После оценки допустимых комбинаций пользователю должен возвращаться список лучших вариантов, доступных для просмотра и сохранения.
Предусматриваются и дополнительные сценарии. Пользователь сможет открыть карточку вещи и изменить ее характеристики, если результаты подбора окажутся неудовлетворительными. Также должна быть возможность удалить некорректную запись, сохранить понравившийся образ и позже вернуться к нему в отдельном разделе. Отдельный сценарий будет связан с просмотром аналитики, чтобы пользователь мог увидеть слабые места своего гардероба.
2.1.3. Потоки входных и выходных данных
Для карточки вещи планируется хранить название, категорию, подкатегорию, цветовые характеристики, стиль, сезон, уровень формальности, материал, а также параметры, влияющие на подбор при разных погодных условиях. Дополнительно должна быть предусмотрена загрузка изображения вещи. Сам файл предполагается сохранять в локальной директории, а в базе данных хранить путь к нему.
Запрос на подбор образа будет содержать другой набор данных. Обязательным параметром планируется сделать тип события. Остальные характеристики смогут задаваться полностью или частично. Пользователь должен иметь возможность указать желаемый стиль, предпочтительные цвета, температуру, погодные условия, ограничения и опорную вещь. Если часть параметров не будет задана вручную, система должна будет дополнить их автоматически.
В качестве результата система должна возвращать список сформированных образов с описанием их состава. Для каждого варианта предполагается передавать итоговую оценку, частные оценки по отдельным признакам и краткое текстовое пояснение. Если подходящие сочетания построить не удастся, система должна вернуть пустой результат и понятное сообщение о причине.
2.1.4. Детализация основного процесса подбора образов
Процесс подбора образов планируется организовать как последовательность нескольких этапов в соответствии с Рисунком 2.2. Сначала клиентская часть будет формировать запрос пользователя. Затем серверная часть должна будет получить сведения о профиле и вещах пользователя. После этого планируется собрать контекст подбора и дополнить его недостающими параметрами. Далее модуль подбора должен сформировать набор допустимых комбинаций. Каждая комбинация будет оцениваться по выбранным признакам. После расчета результаты предполагается отсортировать и сократить до ограниченного числа лучших вариантов. На завершающем этапе система должна сформировать пояснение и вернуть результат пользователю.
Рисунок 2.2 — Блок-схема процесса подбора образа
Отдельное место в процессе проектирования занимает фильтрация некорректных комбинаций. Предполагается, что система не будет оценивать все возможные сочетания без предварительного отбора. Часть вариантов должна исключаться еще до расчета итогового балла. На этапе фильтрации планируется исключать комбинации с отсутствием обязательных категорий, дублированием одной и той же вещи, явным противоречием погодным условиям или другим ограничениям запроса.
2.1.5. Функции подсистемы сохранения и аналитики
После просмотра результатов пользователь должен иметь возможность сохранить выбранный образ. Для этого будет создана отдельную подсистему, которая будет фиксировать состав комплекта и связывать его с конкретным пользователем. В дальнейшем пользователь сможет просматривать сохраненные варианты и использовать их повторно.
Отдельно будет реализована подсистема аналитики. Ее задача будет состоять в обработке данных о гардеробе и формировании кратких выводов о его наполненности. Для этого система должна будет подсчитывать количество вещей по категориям, сезонам и стилевым характеристикам. На основе этих данных пользователю будут выдаваться рекомендации, которые помогут понять, какие элементы гардероба представлены недостаточно и почему некоторые варианты образов формируются с ограничениями.

2.2. Проектирование архитектуры системы
После определения функций системы требуется выбрать такую архитектуру, которая позволит реализовать все основные сценарии работы сервиса. В проекте планируется использовать клиент-серверную архитектуру. Клиентская часть будет отвечать за взаимодействие с пользователем и отображение данных, а серверная часть будет принимать запросы, выполнять проверку входных сведений, запускать логику подбора образов и возвращать результаты. Структурированные данные предполагается хранить в реляционной базе данных, а изображения вещей и фотографии сохраненных образов, в локальном файловом хранилище. Такой подход позволит разделить интерфейс, прикладную логику и хранение данных. 
2.2.1. Выбор архитектурного подхода
Архитектуру веб-сервиса планируется построить на разделении трех уровней: уровня представления, уровня прикладной логики и уровня данных. На стороне клиента будут располагаться страницы, формы и элементы интерфейса, через которые пользователь сможет работать с сервисом. На стороне сервера предполагается разместить обработку запросов, проверку входных данных, модуль подбора образов и доступ к базе данных. Отдельно будет использоваться подсистема хранения, которая объединит базу данных и файловую директорию для изображений.
Выбор такого подхода связан с тем, что он делает структуру системы более упорядоченной. При изменении интерфейса не потребуется перестраивать логику подбора. При доработке алгоритма подбора не потребуется менять базовую организацию пользовательских экранов. Это упрощает развитие проекта и делает архитектуру более удобной для сопровождения.
Рисунок 2.3 — Архитектура клиентской и серверной частей системы
Место для рисунка 2.3. На схеме следует показать браузер пользователя, клиентскую часть, серверную часть, базу данных, файловое хранилище и сервис получения погодных данных. Также важно отразить, что модуль подбора работает внутри серверной части и не вызывается напрямую из интерфейса. 
2.2.2. Архитектура клиентской части
Клиентскую часть планируется организовать как совокупность страниц, повторно используемых компонентов, средств навигации и модулей обмена данными с сервером. Такой подход позволит разделить пользовательские сценарии по отдельным экранам и упростит дальнейшее развитие интерфейса. Отдельные страницы будут отвечать за завершенные действия пользователя, такие как вход в систему, просмотр гардероба, подбор образов, работа с сохраненными комплектами и просмотр аналитики.
Повторно используемые элементы интерфейса предполагается применять для карточек вещей, форм ввода, блоков статистики и карточек образов. Обмен данными с серверной частью также планируется вынести в отдельный слой, чтобы логика отправки запросов не дублировалась в разных частях приложения. Это сделает клиентскую часть более понятной по структуре и удобной для расширения. 
2.2.3. Архитектура серверной части
Серверную часть планируется построить по модульному принципу. Центральный модуль будет отвечать за создание приложения, загрузку настроек, подключение вспомогательных средств и регистрацию функциональных разделов системы. Такое построение позволит постепенно расширять проект без изменения общей структуры.
Внутри серверной части предполагается выделить несколько групп маршрутов. Первая группа будет связана с регистрацией, входом и получением данных профиля пользователя. Вторая группа будет обслуживать операции с карточками одежды и загрузкой изображений. Третья группа будет отвечать за подбор образов, сохранение результатов, загрузку фотографий к сохраненным комплектам и прием пользовательской оценки. Четвертая группа будет формировать сводную статистику по гардеробу.
Также в серверной части важно разделить обработку веб-запросов и прикладную логику. Обработчики запросов должны принимать и разбирать входные данные, вызывать проверку корректности и передавать сведения в сервисный слой. Сервисный слой будет выполнять основную работу, связанную с подбором образов, расчетом оценок, формированием пояснений и построением аналитики. Такое разделение сделает код более упорядоченным и упростит его проверку. 
2.2.4. Взаимодействие компонентов и обмен данными
Обмен между клиентской и серверной частями планируется организовать по модели запрос-ответ. Клиентская часть будет отправлять веб-запросы, получать структурированные ответы и отображать полученные данные в интерфейсе. Для защищенных операций предполагается использовать механизм авторизации, при котором к запросу будет добавляться маркер доступа пользователя.
Отдельно требуется предусмотреть порядок работы с изображениями. Серверная часть должна будет проверять тип загружаемого файла, сохранять его в локальную директорию и заносить в базу данных путь к этому файлу. Такой подход позволит отделить хранение изображений от хранения основной информации о пользователях, вещах и сохраненных образах.

2.3. Проектирование базы данных
Проектирование базы данных необходимо для определения того, каким образом в разрабатываемом веб-сервисе будут храниться сведения о пользователях, карточках одежды, сохраненных образах, предпочтениях и пользовательской реакции на результаты подбора. Для проектируемой системы предполагается использовать реляционную модель данных на базе СУБД MySQL. Такой выбор обусловлен тем, что в сервисе планируется хранить структурированную информацию с фиксированным набором атрибутов, а также устойчивые связи между основными сущностями.
Реляционная модель в данном проекте является оправданной по следующим причинам:
данные имеют выраженную структуру и описываются конечным набором полей;
между объектами сервиса существуют стабильные связи один к одному, один ко многим и многие ко многим;
требуется поддержка ограничений целостности, уникальности и ссылочной связанности;
необходимо обеспечить предсказуемую работу CRUD-операций и алгоритма подбора образов;
база данных должна оставаться расширяемой при подключении внешних сервисов, в частности сервиса определения погоды.
При этом часть многозначных характеристик предполагается хранить в JSON-полях. Это позволит не усложнять схему на раннем этапе разработки и при этом сохранить возможность работы со списками цветов, стилей, ограничений и причин выбора образа.
На текущем этапе в модели данных планируется выделить следующие основные сущности:
пользователь;
предпочтения пользователя;
вещь;
образ;
состав образа;
обратная связь.
Такая структура позволит разделить учетные данные, параметры персонализации, данные цифрового гардероба, результаты подбора образов и реакцию пользователя на эти результаты.
2.3.1. Обоснование выбора модели базы данных
В разрабатываемом сервисе база данных должна поддерживать несколько основных сценариев хранения:
потребуется хранить учетные записи пользователей, поскольку система предполагает регистрацию, вход в личный кабинет и доступ только к собственным данным;
потребуется хранить карточки одежды с характеристиками, которые будут использоваться при подборе образов;
необходимо предусмотреть хранение сохраненных пользователем образов, включая их состав, итоговую оценку и пояснения;
для персонализации рекомендаций потребуется хранить предпочтения пользователя;
потребуется фиксировать пользовательскую реакцию на сохраненные образы.
Из перечисленных требований следует, что проектируемая база данных должна поддерживать:
связь один к одному между пользователем и его предпочтениями;
связь один ко многим между пользователем и вещами;
связь один ко многим между пользователем и сохраненными образами;
связь многие ко многим между образом и вещами через промежуточную сущность;
связь между образом и пользовательской реакцией.
Для описания структуры проектируемой базы данных целесообразно использовать ER-модель. В рамках ER-представления будут выделены сущности, их атрибуты, первичные ключи, внешние ключи и типы связей. Такой подход позволит логически обосновать структуру хранилища до этапа непосредственной реализации и затем без существенных изменений перенести проектную модель в ORM-слой backend-приложения. Погодные параметры рассматриваются как контекст подбора образа, а не как самостоятельный объект предметной области.

2.3.3. Связи и ограничения целостности
В проектируемой модели данных предполагается использовать следующие основные связи:
между пользователем и предпочтениями — связь один к одному;
между пользователем и вещами — связь один ко многим;
между пользователем и сохраненными образами — связь один ко многим;
между образом и составом образа — связь один ко многим;
между вещью и составом образа — связь один ко многим;
между образом и обратной связью — связь один ко многим;
между пользователем и обратной связью — связь один ко многим.
Содержательно связь между сущностями «Образ» и «Вещь» является связью многие ко многим, однако в физической модели она будет реализована через промежуточную таблицу «Состав образа».
Для поддержки корректности проектируемой базы данных планируется использовать первичные ключи, внешние ключи, ограничения уникальности и каскадное удаление связанных записей.
Кроме ограничений на уровне базы данных, на серверной стороне также планируется выполнять прикладную проверку прав владения:
пользователь сможет получать доступ только к своим вещам;
пользователь сможет сохранять только собственные комбинации вещей;
пользователь не сможет сохранить образ, содержащий чужие элементы гардероба;
пользователь сможет просматривать и изменять только свои сохраненные образы.

2.4. Моделирование алгоритма подбора образов
Проектирование алгоритма подбора образов является центральной частью разрабатываемого веб-сервиса, так как данный модуль должен обеспечивать переход от набора разрозненных вещей пользователя к нескольким осмысленным вариантам готового комплекта. В рамках проекта не планируется использовать большие языковые модели, методы компьютерного зрения или непрозрачные рекомендательные модели. Вместо этого предполагается реализовать интерпретируемый rule-based алгоритм, основанный на наборе признаков и системе весов.
Выбор такого подхода обусловлен несколькими причинами:
он позволяет явно задать правила сочетаемости вещей и контролировать качество выдаваемых рекомендаций;
такой алгоритм проще описать, проверить и доработать в рамках дипломного проекта;
он обеспечивает объяснимость результата: система сможет не только выбрать образ, но и показать пользователю, по каким причинам данный вариант оказался удачным.
Предполагается, что модуль подбора образов будет выделен в отдельный backend-компонент, который станет получать данные о вещах пользователя, параметрах запроса и погодном контексте, после чего формировать, оценивать и ранжировать набор возможных комплектов.
2.4.1. Постановка задачи подбора образов
Задача подбора образов в проектируемом сервисе может быть сформулирована следующим образом: по заданному гардеробу пользователя и контексту запроса требуется выбрать несколько наиболее подходящих комплектов одежды, оценить их по набору признаков и вернуть ограниченный список лучших результатов вместе с текстовым пояснением.
На вход алгоритма планируется подавать:
множество вещей, принадлежащих текущему пользователю;
тип события;
предпочтительные цвета;
предпочтительный стиль;
температуру воздуха;
погодное состояние;
ограничения пользователя;
при необходимости идентификатор опорной вещи.
На выходе алгоритм должен возвращать не один вариант, а несколько лучших решений, так как в пользовательском сценарии важнее предоставить человеку выбор между несколькими качественными комплектами. В проектируемой версии сервиса целесообразно ограничить выдачу пятью лучшими образами.

2.4.3. Проектирование этапов алгоритма
Работу алгоритма целесообразно организовать как последовательность независимых этапов. Такое разбиение упростит проверку логики и позволит дорабатывать отдельные части модуля без переписывания всей системы.
Предполагается, что общий поток работы алгоритма будет выглядеть следующим образом:
получение вещей пользователя и параметров запроса;
формирование контекста генерации;
разделение вещей по категориям;
построение кандидатных генераций;
фильтрация некорректных вариантов;
оценка каждого образа по 10 признакам;
расчет итогового балла;
сортировка результатов;
выбор 5 лучших образов;
формирование причин и текстового пояснения.
На первом шаге вещи пользователя планируется разделять по ролям: верх, низ, обувь, верхняя одежда и аксессуары. Далее из этих наборов будут строиться кандидатные комбинации. Базовым вариантом будет считаться комплект вида:
верх + низ + обувь;
верх + низ + обувь + верхняя одежда;
при наличии подходящего аксессуара он также может быть включен в образ как дополнительный элемент.
Если пользователь задаст опорную вещь, алгоритм должен будет учитывать ее как обязательный элемент одного из комплектов. Такой подход позволит формировать образ вокруг уже выбранного предмета гардероба.
После построения кандидатных комбинаций предполагается запуск фильтрации. На данном этапе будут отсекаться варианты:
без обязательных категорий;
с повторным использованием одной и той же вещи;
не соответствующие событию по жестким правилам;
нарушающие ограничения пользователя;
заведомо неподходящие по обуви или верхней одежде для погоды и температуры.
Такое отсечение необходимо, чтобы не передавать в оценочную часть комбинации, которые уже на базовом уровне противоречат логике подбора.
2.4.4. Система оценочных признаков
После фильтрации каждый допустимый кандидатный образ планируется оценивать по десяти признакам. Каждый признак будет принимать значение в диапазоне от 0 до 1, где 0 означает полное несоответствие критерию, а 1 — максимально удачное соответствие.

Признак color_harmony должен оценивать согласованность цветовой палитры. Для этого планируется привести цветовые значения к укрупненным семействам, выделить нейтральные цвета, а затем оценивать образ по типовым цветовым схемам: нейтральная база, монохромная схема, близкие цвета и аккуратный контрастный акцент. При этом обувь и аксессуары целесообразно оценивать мягче, чем верх и низ, так как они могут выполнять роль дополнительного акцента.
Признак style_match должен показывать, насколько вещи сочетаются по стилевому направлению. Для этого предполагается выделить несколько стилевых семейств, например минималистичное, классическое, повседневное, спортивное и вечернее, после чего оценивать согласованность образа на уровне этих семейств.
Признак event_match должен отвечать за соответствие образа сценарию использования. Например, для офисного события планируется повышать оценку комплектам с более сдержанными вещами, а для спортивного — комплектам с уместной спортивной обувью и повседневными вещами.
Признаки season_match, temperature_match и weather_condition_match должны вместе отвечать за климатическую уместность комплекта. В частности, предполагается учитывать:
сезон вещи;
степень утепления;
наличие верхней одежды;
тип обуви;
защиту от дождя и ветра.
Такой набор позволит избежать ситуаций, когда эстетически удачный образ оказывается непригодным для реальных погодных условий.
Признак completeness должен проверять полноту комплекта. Признак layering_correctness — логичность многослойности и отсутствие конфликтов между базовым, промежуточным и верхним слоем. Признаки user_preference_match и constraints_match будут использоваться для персонализации результата.

2.4.6. Формирование итогового результата и пояснения
После вычисления итогового балла все допустимые кандидаты планируется отсортировать по убыванию оценки. Пользователю будут возвращаться только 5 лучших вариантов.
Для каждого результата предполагается формировать:
состав образа;
итоговый балл;
оценки по признакам;
короткий список причин;
связное текстовое пояснение.
Механизм объяснения также целесообразно построить на rule-based основе. Для этого можно выбрать 2–4 признака с наибольшим вкладом и сопоставить им шаблонные формулировки. 
Такой подход позволит сохранить объяснимость и сделать рекомендации более понятными для пользователя.

2.4.8. Перспективы развития алгоритма
Спроектированная модель алгоритма должна оставаться расширяемой. В дальнейшем возможно развитие по следующим направлениям:
расширение набора признаков за счет учета принтов, фактур и контраста;
использование пользовательской обратной связи для уточнения весов;
расширение системы объяснимости;
введение более подробных правил для различных дресс-кодов.
Таким образом, проектируемый алгоритм подбора образов должен представлять собой прозрачную и расширяемую основу, которая уже на первом этапе позволит формировать несколько контекстно уместных комплектов одежды на основе цифрового гардероба пользователя.
2.5. Проектирование пользовательского интерфейса
Проектирование пользовательского интерфейса требуется для того, чтобы определить структуру экранов веб-сервиса, порядок переходов между ними и способ представления основных функций пользователю. Для сервиса подбора образов интерфейс имеет особое значение, так как пользователь работает с визуальными объектами: фотографиями вещей, карточками гардероба, досками образов и сохраненными комплектами.
На этапе проектирования будет создан интерфейс, который будет поддерживать основные сценарии: регистрацию и вход, просмотр главной страницы, управление цифровым гардеробом, добавление и редактирование вещи, подбор образа, сохранение результата и просмотр сохраненных комплектов, аналитические сведения о гардеробе.
2.5.1. Проектирование навигации пользователя
Навигация пользователя должна быть построена так, чтобы основные действия были доступны из личного кабинета без большого числа вложенных переходов. После авторизации пользователь должен попадать на главную страницу с аналитикой. С нее планируется организовать переходы к гардеробу, генерации образов и сохраненным комплектам.

2.5.2. Страница входа
Рисунок 2.6 — Прототип страницы входа
Страница входа предназначена для авторизации пользователя. На ней планируется разместить форму с двумя основными полями: адрес электронной почты и пароль. После успешного входа пользователь будет перенаправляться на главную страницу личного кабинета.
Если учетной записи еще нет, на странице входа должен быть предусмотрен переход к регистрации. Такой сценарий позволит пользователю быстро выбрать нужное действие без поиска дополнительного меню.
2.5.3. Страница регистрации
Страница регистрации должна использоваться для создания учетной записи. В форму планируется включить имя пользователя, город, email и пароль. 
Рисунок 2.7 — Прототип страницы регистрации
После успешной регистрации пользователь должен получить доступ к личному кабинету. При наличии учетной записи пользователь сможет перейти обратно на страницу входа.
2.5.4. Главная страница с аналитикой
Главная страница будет выполнять роль стартовой панели пользователя. На ней планируется разместить приветственный блок, краткую информацию о гардеробе, быстрые переходы к основным разделам и аналитические показатели.
Так как отдельная вкладка аналитики в дальнейшем не планируется, основные аналитические данные целесообразно вынести именно на главную страницу. В этом блоке пользователь сможет увидеть общее количество вещей, распределение по категориям, сезонам и стилям, а также короткие рекомендации по наполнению гардероба.
Рисунок 2.8 — Прототип главной страницы с аналитикой
С главной страницы пользователь должен переходить к следующим действиям:
открыть гардероб;
добавить новую вещь;
перейти к подбору образа;
открыть сохраненные образы.
2.5.5. Страница гардероба
Страница гардероба предназначена для просмотра всех вещей пользователя. Основной элемент экрана — сетка карточек. Каждая карточка должна содержать фотографию вещи, название и две иконки действий: редактирование и удаление.
Рисунок 2.9 — Прототип страницы гардероба
Полные характеристики вещи не планируется показывать прямо в списке, чтобы не перегружать страницу. При нажатии на карточку пользователь должен переходить на отдельную страницу вещи. Такой подход позволяет сохранить список компактным и при этом оставить доступ к подробной информации.
Со страницы гардероба пользователь сможет:
открыть карточку вещи;
перейти к созданию новой вещи;
перейти к редактированию выбранной вещи;
удалить вещь из гардероба.
2.5.6. Страница карточки вещи
Карточка вещи должна показывать полную информацию о выбранном предмете гардероба. В первой колонке планируется разместить изображение вещи. Во второй колонке — название, категорию, подкатегорию, сезон, цвета, стили, формальность, посадку, слой, утепление и погодные признаки.
Рисунок 2.10 — Прототип карточки вещи
Кнопки действий целесообразно оформить как компактные иконки. Они должны располагаться в нижней части информационной колонки. Это позволит отделить основные сведения от действий редактирования и удаления.
Эта страница должна открываться из гардероба и из карточки образа. Такой переход важен, потому что пользователь может увидеть вещь в составе образа и сразу перейти к ее описанию.
2.5.7. Страница создания и редактирования вещи
Форма создания и редактирования вещи должна обеспечивать контролируемый ввод характеристик. Для большинства полей планируется использовать выпадающие списки, переключатели и множественный выбор. Цвета целесообразно показывать в виде цветных кружков, чтобы пользователь выбирал их визуально. 
Такой подход нужен для однородности данных. Если пользователь будет вводить характеристики произвольным текстом, алгоритму подбора будет сложнее сравнивать вещи между собой. Поэтому интерфейс формы должен направлять пользователя к выбору из заранее подготовленных значений. 
В форме планируется предусмотреть:
загрузку изображения;
название вещи;
категорию и подкатегорию;
цвета;
стили;
сезон;
формальность;
посадку;
уровень слоя;
утепление;
признаки защиты от дождя и ветра.

2.5.8. Страница подбора образов
Страница подбора образов должна использоваться для задания параметров генерации. Пользователь сможет выбрать тип события, предпочтительный стиль, желаемые цвета, погодные условия, температуру, ограничения и, при необходимости, опорную вещь. 
После запуска подбора система должна показать пользователю результаты в визуальном формате. Вместо обычного списка планируется использовать карточку-доску образа. На такой доске вещи будут представлены как отдельные изображения, а рядом или в отдельном блоке будет отображаться оценка и краткое пояснение.

2.5.9. Карточка нового образа
Карточка нового образа должна открываться как отдельное визуальное окно поверх страницы подбора. В ней планируется разместить доску с вещами, итоговую оценку, краткое пояснение и причины выбора.
На доске элементы одежды должны располагаться равномерно и не перекрывать друг друга.

сохранить образ;
закрыть карточку;
открыть страницу конкретной вещи.
2.5.10. Страница сохраненных образов
Страница сохраненных образов предназначена для повторного просмотра комплектов, которые пользователь отметил как удачные. Она должна содержать список сохраненных карточек и давать возможность открыть каждый образ подробнее. 
На этой странице пользователь сможет вернуться к ранее выбранному комплекту, оценить его состав и при необходимости перейти к вещам, входящим в образ. Такой раздел нужен для того, чтобы результат подбора не исчезал после закрытия страницы генерации.

Во второй главе были рассмотрены проектные решения, которые определяют структуру веб-сервиса подбора образов и управления личным гардеробом. Были выделены основные функции системы: регистрация и вход пользователя, ведение цифрового гардероба, автоматический подбор образов, сохранение удачных комплектов и формирование краткой аналитики по составу гардероба.
В ходе проектирования была описана функциональная схема сервиса и определён основной сценарий взаимодействия пользователя с системой. Для реализации сервиса была выбрана клиент-серверная архитектура с разделением на уровень представления, уровень прикладной логики и уровень данных. Клиентская часть отвечает за интерфейс и ввод данных, серверная часть обрабатывает запросы, проверяет сведения, запускает модуль подбора и возвращает результаты. Для хранения структурированных данных выбрана реляционная база данных MySQL, а для изображений вещей и фотографий сохранённых образов предусмотрено файловое хранилище.
В главе была спроектирована структура базы данных. Были определены сущности пользователя, предпочтений, вещи, образа, состава образа и обратной связи. 
Также был разработан алгоритм подбора образов. В качестве основы выбран rule-based подход, который позволяет явно задавать правила сочетаемости, контролировать результат и формировать понятные пояснения для пользователя. 
Отдельно был спроектирован пользовательский интерфейс веб-сервиса. Были определены основные экраны: вход, регистрация, главная страница с аналитикой, гардероб, карточка вещи, форма создания и редактирования вещи, страница подбора образов, карточка нового образа, сохранённые образы и карточка сохранённого образа. Структура интерфейса поддерживает все основные пользовательские сценарии и делает работу с сервисом последовательной.
В результате выполнения второй главы была сформирована проектная основа для дальнейшей реализации веб-сервиса. Были определены функции системы, архитектура, структура базы данных, логика алгоритма подбора и состав пользовательских экранов.
